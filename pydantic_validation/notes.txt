Request Body: A request body is the data sent by the client to the server.

React App  --> POST /users --> Body {...} --> 

| HTTP Method | Request Body? |
| ----------- | ------------- |
| GET         | Usually No    |
| POST        | Yes           |
| PUT         | Yes           |
| PATCH       | Yes           |
| DELETE      | Sometimes     |

Why not use query parameters?
/signup?name=Azeem&email=azeem@gmail.com&password=123456

Problems:
- Password is visible in the URL.
- Limited amount of data can be sent.
- Difficult to send complex structures.

Instead, the browser sends the information in the request body.
{
    "name": "Azeem",
    "email": "azeem@gmail.com",
    "password": "123456",
    "age": 22
}


REST API Convention
A backend engineer chooses the HTTP method based on the operation:
| Operation               | HTTP Method          |
| ----------------------- | -------------------- |
| Get all users           | `GET /users`         |
| Get one user            | `GET /users/{id}`    |
| Create a user           | `POST /users`        |
| Update a user           | `PUT /users/{id}`    |
| Partially update a user | `PATCH /users/{id}`  |
| Delete a user           | `DELETE /users/{id}` |

--> A request body carries structured data from the client to the server.
--> FastAPI uses Pydantic models to describe and validate that data.
--> BaseModel is the foundation of every Pydantic model.
--> FastAPI automatically parses JSON into Python objects.
--> Validation happens before your route function executes.

Use a separate Pydantic model for each type of request.
Give models meaningful names (UserCreate, ProductCreate, LoginRequest) rather than generic names like Data.

Field() lets you attach:
- validation rules
- documentation
- examples
- descriptions
- default values
- constraints

Instead of using plain str for everything, Pydantic offers specialized types
_____________________________________________________
| Type       | Purpose                              |
| ---------- | ------------------------------------ |
| `EmailStr` | Email validation                     |
| `AnyUrl`   | URL validation                       |
| `UUID`     | UUID values                          |
| `datetime` | Date & time                          |
| `date`     | Calendar date                        |
| `Decimal`  | Precise decimal values (e.g., money) |

Note: EmailStr requires an additional package: pip install email-validator

Nested Models: Production APIs often work with nested objects.

class Address(BaseModel):
    city: str
    country: str
class Student(BaseModel):
    name: str
    address: Address
Request:
{
    "name": "Ali",
    "address": {
        "city": "Lahore",
        "country": "Pakistan"
    }
}
Pydantic validates both models automatically.

Lists of Models: Suppose an order contains multiple products.
class Product(BaseModel):
    name: str
    quantity: int
class Order(BaseModel):
    customer: str
    products: list[Product]
Request:
{
    "customer": "Ali",
    "products": [
        {
            "name": "Laptop",
            "quantity": 1
        },
        {
            "name": "Mouse",
            "quantity": 2
        }
    ]
}
Every product in the list is validated individually.
